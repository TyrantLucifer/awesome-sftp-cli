# ADR-0002：客户端与持久守护进程分离

- 状态：Accepted
- 日期：2026-07-14
- 影响范围：进程模型、本机 RPC、会话与任务所有权、外部程序启动

## 背景

产品既是交互式双栏 TUI，也是可能运行数小时的传输与搜索工具。用户需要退出界面、重新打开终端、进入 Vim 或 shell，而数百 GB 文件和百万级目录任务不能因此丢失。把全部状态留在 TUI 进程会把终端生命周期和数据操作生命周期绑定在一起；为每个任务启动独立命令又会造成会话、队列、冲突和缓存状态分裂。

## 决策

采用本机客户端/守护进程模型。一个 Go 可执行文件以不同角色运行：

- TUI/CLI 客户端：输入、展示、确认、冲突决策与外部程序编排。
- 本机守护进程：session、workspace、job、cache、provider 和持久化的唯一所有者。
- SSH_ASKPASS broker 入口：只处理单次受限认证 challenge。
- 同一 Go 程序的目标平台 helper 角色是远端能力组件，不属于本机常驻服务；它按远端 OS/架构单独构建，但不形成第二套代码库或产品。

客户端通过权限为 0600 的 Unix domain socket 使用版本化长度分帧 JSON RPC。守护进程未运行时，客户端可以安全自动启动它；并发启动由单实例锁与套接字所有权判断收敛。

## 职责边界

### 客户端拥有

- pane 的瞬时渲染状态、键序列和弹窗焦点
- Vim 风格意图的生成
- 高风险确认、冲突选择和认证回答
- 暂停 TUI 后启动 Vim、默认打开器或交互 shell
- 从快照和事件重建 UI

### 守护进程拥有

- endpoint session 与能力探测
- workspace 可恢复状态
- 不可变 Plan 与 Job 状态机
- transfer、search、preview、cache 任务
- SQLite 事务、检查点和 schema migration
- Auth Broker challenge
- 并发、速率、重试和资源预算

客户端断开只是订阅者消失，不是 Job 取消信号。只有显式 CancelJob 才触发取消。

## RPC 契约

- 首次握手协商协议主次版本和事件能力。
- 查询返回一致性快照；长操作返回持久 Job ID 或 Search ID。
- 服务端事件携带序号与相关 ID。重连时先获取快照，再续接仍保留的事件。
- 状态变更请求带 request ID，在守护进程的有限去重窗口内最多生效一次。
- 客户端必须能处理未知的可选字段与事件类型；主版本不兼容时明确拒绝。
- RPC 错误包含稳定类别、可读上下文与是否可重试，不以日志字符串作为控制流。

## 外部程序

TUI 不内嵌终端模拟器：

- e 启动 Vim 或 EDITOR 编辑缓存租约文件。
- o 使用系统默认打开器打开缓存租约文件。
- 感叹号执行一次性、用户明确输入的当前位置命令。
- gs 暂停 TUI 并进入本地或远端交互 shell。

守护进程不把任意 shell 当作 provider API。文件回传仍通过标准 Job 管线完成。外部程序退出、客户端崩溃或 lease 超时都由缓存恢复逻辑处理。

## 持久化与恢复

守护进程在 SQLite 中事务保存 Job、Plan、检查点、workspace 和缓存 lease。每次从 running 边界恢复时都检查实际文件后置条件，不能简单把进程前的内存状态当成事实。

客户端重连后：

1. 完成版本握手。
2. 获取 workspace、pane、session、job 和 challenge 快照。
3. 订阅新事件。
4. 使用事件序号消除快照与订阅之间的竞态。

守护进程崩溃后由下一次客户端启动或平台服务机制重启；恢复审计决定任务继续、等待认证、等待冲突或失败。

## 安全边界

- 守护进程只以当前用户运行，不监听 TCP，不要求 root。
- 套接字位于当前用户私有目录并使用 0600；支持时校验 peer UID。
- 单实例判断不能通过跟随不可信符号链接覆盖其他文件。
- RPC 不传输持久凭据；认证回答绑定一次性 challenge。
- 外部程序只获得本地文件路径和必要环境，不获得 RPC capability token 或远端认证材料。
- 日志和导出诊断在守护进程侧统一脱敏。

## 降级与失败行为

- 守护进程不可达：客户端尝试一次安全启动；失败后进入只显示诊断的错误页，不偷偷启动第二个状态库实例。
- RPC 次版本差异：按能力隐藏新功能；主版本差异：拒绝写操作并提示匹配版本。
- 客户端退出：后台 Job 继续；需输入的任务进入 waiting_auth 或 waiting_conflict。
- 数据库不可写或迁移失败：停止新写任务，保留只读诊断能力；不覆盖原库。
- 外部编辑器或打开器失败：释放或保留 lease 按错误阶段决定，远端文件不变。

## 替代方案

### 单一 TUI 进程

实现简单，但退出、崩溃和外部程序切换都会威胁长任务；多个窗口也无法共享会话和队列，因此拒绝。

### 每个任务独立后台命令

进程隔离清晰，但冲突、认证、带宽、缓存和恢复状态难以统一，且会重复建立 SSH 会话，因此拒绝。

### 常驻 GUI 或内嵌终端

能容纳外部程序，但偏离终端优先的产品边界，并引入平台 UI 与终端模拟复杂度，因此不在首版采用。

### 多个独立可执行文件

职责可见，但发布、版本匹配和安装更复杂。使用同一二进制的角色入口，仍在代码层保持边界。

## 后果

### 正面

- 长任务独立于终端和 TUI 生命周期。
- 多个客户端可共享只读状态与任务视图，写操作由一个事实源串行约束。
- session、cache、auth 和资源调度集中管理。

### 代价

- 必须维护 RPC 兼容、daemon 生命周期、单实例和迁移。
- UI 是最终一致视图，需要处理重连、迟到事件和过期加载结果。
- 测试必须覆盖进程级故障，而不仅是包内单元测试。

## 验证要求

- 并发客户端启动只产生一个守护进程和一个可写状态库。
- TUI kill -9 后 Job 继续；daemon kill -9 后按检查点恢复且不破坏目标。
- RPC 分帧、版本、去重、事件续接与恶意长度输入的 contract/fuzz 测试。
- macOS 与 Linux 的 Vim、默认打开器和 shell 暂停/恢复 smoke 测试。
