# ADR-0004：崩溃安全的传输语义

- 状态：Accepted
- 日期：2026-07-14
- 影响范围：copy、move、编辑回传、覆盖、删除、恢复和 direct 降级

## 背景

目标场景包含不稳定网络、客户端退出、守护进程 kill -9、数百 GB 文件、跨两个远端移动和并发修改。若直接写最终目标，失败会留下看似完整的损坏文件；若把 move 当成一条不可分命令，目标状态不明确时可能误删来源；若对所有失败自动重试，rename 或 delete 等非幂等操作可能重复执行。

## 决策

所有写操作采用 Intent、Plan、Job 三层模型，并遵守“临时写入、验证、原子提交”的目标协议。move 被定义为 copy、verify、commit、delete-source 四阶段组合，来源删除是最后且独立审计的步骤。

目标默认写为与最终文件同目录的 .part-<job-id>。写入与可恢复检查点完成后进入 verifying；满足冻结 Plan 的验证策略后，以 provider 支持的原子 rename 提交为最终名。

## 计划阶段

Planner 在入队前冻结：

- 来源 FileRef 与版本指纹
- 最终目标及冲突策略
- 临时目标命名和现有临时文件处理
- 路由与允许的降级
- 偏移恢复、稀疏文件和符号链接策略
- 验证算法和最低强度
- 是否为 move，以及删除来源的前置条件
- 覆盖、trash、权限和元数据策略
- 并发、带宽、缓存与内存预算

无冲突普通 copy 可直接 queued。overwrite、不可逆 delete 和 direct 必须 awaiting_confirmation。apply-to-all 冲突决定只绑定当前 Job，不能成为隐式全局设置。

## 状态机

主路径：

    draft
      -> awaiting_confirmation
      -> queued
      -> running
      -> verifying
      -> completed

等待与控制状态：

- paused
- waiting_auth
- waiting_conflict
- retry_wait

终止状态：

- completed
- completed_with_source_retained
- failed
- canceled

每次状态迁移与其检查点在同一 SQLite 事务提交。只有领域状态机可以改变状态；worker 通过事件请求迁移，不能直接写任意状态。

## 写入与检查点

- 每个分段检查点记录已确认的目标偏移、来源版本指纹、临时目标身份和必要的流校验状态。
- 恢复前重新 stat 来源与临时目标；与检查点不一致时进入 waiting_conflict 或从安全边界重建，不能仅相信数据库。
- 断点续传只追加到已验证前缀。远端不支持可靠随机写或临时目标身份变化时，从头写新的临时文件。
- 本机中继使用固定上限的内存缓冲，不默认落本地磁盘；背压从目标写端传播到来源读端。
- 取消默认保留可恢复 .part 和检查点。用户可以显式清理，清理本身作为可审计操作执行。

## 验证与提交

验证强度由能力决定，并在 Plan 中可见：

1. 两端完整强哈希
2. 传输流哈希加目标端强哈希
3. 大小、精确修改时间和可用身份组合
4. 仅大小或其他弱验证

系统不得把弱验证描述为完整性证明。对 move，来源删除策略可以要求高于 copy 成功的最低验证强度。

提交前再次检查最终目标冲突。若目标从计划后发生变化，进入 waiting_conflict。支持原子 rename 时，以它作为 commit 点；不支持时必须在计划中明确较弱语义并要求相应确认，不能假装原子。

## Move 与删除

move 的顺序固定为：

1. 复制到目标临时文件。
2. 验证目标。
3. 提交最终目标。
4. 重新确认来源仍是计划中的对象。
5. 删除来源。

在步骤 4 或 5 无法证明安全时，不回滚已经正确提交的目标，也不猜测删除；结果为 completed_with_source_retained。目录 move 按子项结果汇总，任何未验证子项都阻止对应来源删除。

显式 delete 优先使用 provider trash 能力。纯 SFTP 没有 trash 时展示不可逆警告，并在执行前重新核对 FileRef 指纹。

## 路由语义

计划器按以下顺序选择：

1. 同 endpoint 原子 rename、copy-data 或 helper 同主机复制。
2. 预检通过且用户确认的跨主机 direct。
3. 守护进程有界内存中继。

路由优化不得改变目标提交和 move 删除语义。direct 在 commit 前失败且 Plan 允许时可以降级到中继；一旦 commit 状态不明确，必须先审计远端实际状态，禁止立即重跑。

## 重试规则

- list、stat、未提交的有偏移读写等可证明幂等阶段可自动退避重试。
- rename、commit、delete 和远端服务端复制完成信号丢失时，先检查后置条件。
- auth_required 进入 waiting_auth，不作为普通网络错误循环重试。
- conflict 进入 waiting_conflict，不自动覆盖。
- permission_denied 和 integrity_failed 默认不重试，除非用户改变计划或环境。
- 每次尝试保留类别、时间、路由和检查结果，诊断中可见。

## 崩溃恢复

守护进程启动后对非终止 Job 执行恢复审计：

1. 读取冻结 Plan 和最后事务检查点。
2. stat 来源、临时目标和最终目标。
3. 判断 commit 是否已完成、临时文件是否匹配、来源是否仍相同。
4. 从可证明的幂等阶段恢复，或进入 waiting_auth/waiting_conflict。
5. 对状态不明的 delete/rename 不重放。

客户端 kill -9 不改变 Job。守护进程 kill -9 后，数据库与实际文件系统的后置条件共同决定恢复，不依赖上次内存进度。

## 安全边界

- 不在验证前删除来源。
- 不直接覆盖最终目标写入长流。
- 不因重试绕过确认、冲突或认证。
- 不把 .part 文件隐藏成完整结果；UI 与日志必须能关联它的 Job 和清理策略。
- direct 不改变凭据边界，不能通过复制密钥或默认转发 agent 达成。
- 文件名通过 provider 协议传输，不拼入 shell。

## 降级与失败行为

- 无强哈希：使用 Plan 允许的较弱验证；move 不满足删除门槛时保留来源。
- 无原子 rename：计划标为弱提交语义并确认；若策略不允许则失败。
- 临时文件不可创建：不回退为直接写最终文件。
- 目标空间不足或 short write：保留安全临时状态，Job failed 或 retry_wait；最终目标不变。
- 网络断开：从最后验证前缀恢复；无法验证则重建临时文件。
- direct 失败：提交前可降级到中继；提交不明则先审计。
- 来源在传输中变化：禁止删除来源，按策略冲突、重试新版本或以保留来源结果结束。

## 替代方案

### 直接写最终目标

省去 rename 和临时空间，但断线会产生外观正常的部分文件，无法接受。

### 将 move 视为单条复制后删除命令

实现简单，却无法表达验证不足、来源并发变化和删除结果不明，因此拒绝。

### 对任何错误从头自动重试

会重复非幂等 commit/delete，也浪费超大文件传输，因此采用分阶段幂等与后置条件检查。

### 默认本地落盘再上传

恢复直观，但双远端大文件需要额外本地空间并泄露数据副本。默认使用有界内存中继；只有明确缓存或用户策略允许时才落盘。

## 后果

### 正面

- 网络、客户端和 daemon 故障不会把部分内容冒充完整目标。
- move 在所有路由上保持“先证明目标，再删除来源”的一致语义。
- 超大文件可以有界内存、可恢复执行。

### 代价

- 目标需要临时空间，某些 provider 的弱 rename 能力必须显式处理。
- 状态机、后置条件审计和 conflict UI 比简单复制复杂。
- 强验证可能需要额外读取，性能策略必须对用户透明。

## 验证要求

- short write、网络半关闭、disk full、permission denied、stale stat 和目标并发创建。
- 客户端与 daemon 在 running、verifying、commit 和 delete 边界 kill -9。
- 100GB sparse 文件的有界内存传输、取消、恢复和校验。
- 跨远端 move 在目标未验证时绝不删除来源。
- direct 与本机中继产生相同可观察结果和错误分类。
