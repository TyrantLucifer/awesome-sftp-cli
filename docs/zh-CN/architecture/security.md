# 安全设计

[English](../../architecture/security.md)

AMSFTP 会在不同文件系统之间传输数据、启动 OpenSSH，并在本地保存可恢复状态。
因此，它的安全模型强调清晰的责任边界：SSH 信任和认证由 OpenSSH 负责，持久操作
由本地 daemon 负责，每个破坏性动作和外部程序操作由用户决定。

本文说明设计希望提供的保证、这些保证依赖的假设，以及用户需要理解的限制。组件
关系见[架构](overview.md)。

## 信任边界

| 边界 | AMSFTP 信任什么 | 仍需校验什么 |
| --- | --- | --- |
| 本地操作系统账号 | 已登录用户主动运行客户端 | 环境变量、配置、终端输入、文件系统所有者和权限 |
| 客户端与 daemon | 通过同用户和协议校验后的对端 | 帧大小、请求身份、版本、速率、截止时间和消息结构 |
| 系统 OpenSSH | 经过验证的 `/usr/bin/ssh` 及其既有安全策略 | 可执行文件路径、结构化参数、进程生命周期和有界输出 |
| 远端 SSH/SFTP 服务器 | 仅在 OpenSSH 接受主机和认证策略后信任 session 身份 | 每个名称、元数据值、内容字节、能力声明、错误和断线 |
| 编辑器、打开程序与 shell | 仅信任用户明确配置或输入的程序与命令 | 参数结构，以及重新带回 AMSFTP 的数据 |
| 已安装发行版本 | 从文档渠道获得、并按该渠道要求验证的字节 | 安装路径所有者、权限、符号链接和运行时兼容性 |

认证可以证明按 OpenSSH 策略连接到了哪个远端账号，但不代表远端文件系统输出的
内容可以不经检查就直接显示、保存或解释。

## SSH 配置与凭据

AMSFTP 不实现自己的 SSH 协议栈。SFTP provider 使用固定参数启动经过验证的系统
OpenSSH，并通过标准输入输出与 SFTP subsystem 通信。主机别名作为数据传入，不会
经过 shell。

以下内容仍由 OpenSSH 负责：

- 主机密钥验证和 `known_hosts`；
- `~/.ssh/config`、`Include` 与 `Match`；
- 私钥、安全密钥和 SSH Agent；
- Kerberos/GSSAPI；
- 交互式认证；
- `ProxyJump` 与 `ProxyCommand`。

AMSFTP 不会把私钥、Agent 内容、Kerberos 票据、密码或认证回答复制到配置、数据库、
工作区、Job、日志、缓存元数据或支持包中。交互回答只在当前经过验证的认证会话中
存在；没有可信客户端可以回答时，操作会进入等待。

AMSFTP 不会为了兼容性或性能而开启 Agent forwarding、凭据委托，或放宽主机密钥
检查。如果你的 SSH alias 主动启用了这些能力，它们仍属于用户明确配置的 OpenSSH
策略。

## 本地 daemon 与私有状态

daemon 是当前用户专用进程。它使用仅供所有者访问的 Unix socket，而不是网络监听
端口；接受请求前会检查对端用户。协议握手会在分发操作之前拒绝不兼容客户端。

配置、状态、缓存、日志、socket 和升级锁都位于私有根目录中。AMSFTP 使用敏感路径
前会检查预期的所有者、mode、系统支持时的 ACL，以及符号链接情况。根目录不安全
时会直接拒绝，而不是通过放宽权限自动“修复”。

这些检查可以避免不同用户意外共享 AMSFTP 状态，也能阻止常见的路径替换攻击。但
它们不会把 daemon 与同一操作系统账号隔离。已经可以完整控制该用户（或 root）的
进程，通常也能读取该用户文件并影响其会话。

## 远端数据是不可信输入

即使连接已经认证，远端文件名、元数据、文件内容、服务器扩展、错误文本和响应时序
仍按不可信输入处理。

AMSFTP 会：

- 把原始路径字节和显示文本分开；
- 渲染前转义终端控制内容；
- 使用结构化进程参数，不从文件名拼接 shell 命令；
- 相对于预期 provider 边界验证路径；
- 限制帧、名称、列表、预览、搜索和进程输出；
- 只有当前 session 实际证明某项能力后，才使用对应的可选行为。

恶意远端服务器仍然可以返回虚假的内容或元数据，也可以拒绝服务。SSH 会按照
OpenSSH 策略保护传输并确认服务器身份，但无法让已经被入侵的服务器变得可信。

## 安全修改与恢复

客户端不能直接修改 provider。它把操作意图发送给 daemon，由 daemon 把来源、
目标、路由、冲突策略和所需确认冻结为持久 Job。

流式复制会把目标写入 Job 专属临时名称。AMSFTP 只在持久化边界记录进度，随后验证
结果，最后发布最终名称。未完成的传输不会以预期最终文件的身份出现。

移动还有一条额外规则：只有目标已经验证并提交，才会删除来源。如果来源删除结果
不确定，保留一份多余副本比冒险丢失数据更安全。

覆盖、递归删除、冲突处理和其他破坏性选择都要求明确操作。重试只跨越已知幂等的
步骤，或先检查前一次操作是否已经生效。网络中断或 daemon 重启都不足以成为盲目
重复重命名、提交或删除的理由。

这些保证可以降低意外丢失的风险，但不能替代备份。服务器端变更、磁盘故障、账号
被入侵，以及用户主动确认了错误操作，仍可能造成数据损失。

## 缓存、编辑器、打开程序与 shell

远端预览和编辑内容会准备到有配额、有租约的私有缓存中。AMSFTP 已知某个编辑器、
打开程序或预览正在使用条目时，不会回收它。上传编辑结果前，AMSFTP 会比较初始
远端身份和当前远端对象，把并发修改报告为冲突。

编辑器和打开程序以当前操作系统用户权限运行，可以读取交给它们的文件。交互式本地
或远端 shell 是用户明确离开结构化文件操作界面的入口，其中命令拥有对应本地或
SSH 账号的权限；AMSFTP 无法把一条危险命令变成安全命令。

只配置你信任的程序。除非已经检查过引用和行为，否则不要使用会再次经由 shell
传递文件名的命令。

## 诊断与支持包

面向人类和 JSON 的错误只暴露稳定、有界字段，不直接显示底层原始原因。持久日志
只允许经过审核的字段，不记录原始路径、命令参数、环境变量、文件内容或认证材料。

`doctor` 是只读诊断命令，不会修复数据库、修改 SSH 信任、安装远端软件或上传结果。

支持包分两步生成：

1. 预览准确的文件清单、敏感级别和 consent digest；
2. 使用匹配的 digest，把归档写入一个新的用户私有文件。

AMSFTP 不提供自动上传命令。脱敏可以减少暴露，但不会让支持包变成公开信息：它仍
可能包含版本、能力结果、假名化标识和其他系统元数据。分享前请自行检查。

## 资源与拒绝服务边界

目录项、事件历史、日志、搜索结果、预览字节、缓存用量、传输缓冲、连接数、队列和
并发 worker 都有上限。

达到限制时，AMSFTP 会施加背压、等待、返回部分结果或报告资源错误。这样可以避免
本地会话无界增长，但不能阻止远端服务器缓慢响应、反复断线，或占用用户明确请求的
工作所需带宽。

## 可选远端增强

标准 SFTP 是受支持的基线。当前公开构建没有开启远端 Helper 或远端到远端直传。
除非存在另一个明确受支持的安全路由，两个远端之间的复制会使用有界本地中继。

不要为了强行开启禁用路径而安装测试产物、注入信任密钥、委托凭据或削弱主机密钥
策略。可选增强失败时，标准 SFTP 路径必须保持独立可用。

## 用户需要承担的责任

AMSFTP 依赖用户：

- 保护本地账号、终端会话、SSH 配置、Agent 和 Kerberos 票据缓存；
- 通过带外渠道验证意外发生变化的主机密钥；
- 只从文档说明的渠道安装，并完成对应的 checksum 或包管理器验证；
- 仔细检查来源、目标、冲突、覆盖、移动和删除确认；
- 为重要数据保留独立备份；
- 只配置可信的编辑器、打开程序和 shell 命令；
- 保护私有状态和支持包；
- 及时安装 AMSFTP、OpenSSH、操作系统和远端服务器的安全更新。

checksum 只能证明下载字节与发布的 checksum 一致，本身不能替代代码签名、公证，
也不能证明发布者账号未被入侵。各渠道当前提供的保证见
[安装说明](../user/installation.md)。

## 安全模型的边界

AMSFTP 不声称可以抵御：

- 已被入侵的本地用户账号、root、内核、终端、OpenSSH 可执行文件或配置的外部
  程序；
- 已被入侵的远端账号，或返回恶意、虚假数据的服务器；
- 已记录 AMSFTP Job 之外的数据损失；
- 用户主动覆盖 OpenSSH 策略或确认破坏性操作；
- 磁盘耗尽、网络故障或恶意服务器环境中的无限可用性；
- 原生 Windows、GUI 沙箱、远端文件系统挂载或通用凭据保险箱。

在 macOS 上，修改 Gatekeeper 设置前，请从安装说明和发行说明确认当前签名、公证
状态。对于无法独立确认来源和 checksum 的产物，不要关闭操作系统安全控制。
