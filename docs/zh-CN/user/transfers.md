# 传输和 Job

[English](../../user/transfers.md)

AMSFTP 会把复制、移动、重命名和删除都作为可持久恢复的后台 Job 执行。文件栏负责
收集你的选择和确认，daemon 负责记录并执行工作。关闭 TUI 不会静默取消任务。

## 复制文件

1. 使用 `Space`、`v` 或 `V` 选择一个或多个条目。没有选择时会使用光标所在
   条目。
2. 按 `y`。这一步只记录文件引用，不会开始读取数据。
3. 按 `Tab` 切换到目标栏。
4. 按 `p`。
5. 检查来源、目标、条目数和可能的冲突选择，然后确认。
6. 按 `J` 查看新建的 Job。

每个目标都会先写到私有临时内容。AMSFTP 完成验证和提交后，最终名称才会出现。
失败的传输不会被报告成完整的最终文件。

目录复制采用增量遍历。符号链接会作为符号链接显示；递归复制目录时不会沿它进入
目标。

## 移动文件

流程与复制相同，只需把 `y` 换成 `d`：

1. 选择来源。
2. 按 `d`。
3. 切换到目标栏并按 `p`。
4. 检查移动确认，再启动 Job。

小写 `d` 只会把选择标记成剪切，本身不会删除任何内容。

同一个端点支持安全重命名时，AMSFTP 可以直接使用它。否则移动会按照复制、验证、
提交、重新检查原文件、删除原文件的顺序执行。如果原文件已经变化，或者无法证明
删除成功，Job 会保留已经验证的目标，并以“来源已保留”结束。这是有意的安全行为，
需要你检查两份文件。

## 重命名单个条目

把光标放在一个文件或目录上，按 `r`，输入一个普通的新条目名称并确认。重命名不
接受多选。

它仍然会作为 Job 出现，因此执行结果和冲突都可见。

## 删除文件

大写 `D` 是唯一的删除入口，与小写 `d` 完全分开。

1. 选择要删除的文件。
2. 按 `D`，检查已经冻结的选择。
3. 确认范围。
4. 如果端点不支持可靠的回收站，再检查不可逆警告并确认一次。

当前本地和标准 SFTP 通道都没有声明可靠回收站，因此删除通常需要两次确认。
AMSFTP 会拒绝根目录、空目标、已经变化的目标和其他不安全对象。递归删除不会沿符号
链接进入它指向的位置。

## 解决目标冲突

可以选择：

- **覆盖**：确认记录的目标状态没有变化后才替换它。
- **跳过**：保持当前来源和目标不变。
- **自动重命名**：生成一个不冲突的目标名称。
- **询问**：暂停 Job，等待你选择以上一种处理方式。

在 Job 抽屉中：

| 按键 | 作用 |
| --- | --- |
| `w` / `x` / `a` | 覆盖 / 跳过 / 自动重命名当前冲突 |
| `W` / `X` / `A` | 把同一选择应用到此 Job 中匹配的冲突 |

“全部应用”只限于当前 Job，不会变成全局默认值。

## 查看和控制 Job

按 `J` 打开 Job 抽屉，使用 `j`、`k` 选择任务。

| 按键 | 可用时的作用 |
| --- | --- |
| `P` | 暂停排队中或正在运行的任务 |
| `U` | 恢复暂停任务，或重试可恢复的等待 |
| `C` | 取消尚未结束的任务 |

标题只会显示当前状态真正允许的动作。已完成、失败和取消的 Job 仍然可以选择，但
不会显示不存在的操作。

命令行提供相同的控制能力：

```sh
amsftp job list
amsftp job events job_aaaaaaaaaaaaaaaaaaaaaaaaaa --after 0 --limit 50
amsftp job pause job_aaaaaaaaaaaaaaaaaaaaaaaaaa
amsftp job resume job_aaaaaaaaaaaaaaaaaaaaaaaaaa
amsftp job cancel job_aaaaaaaaaaaaaaaaaaaaaaaaaa \
  --confirm job_aaaaaaaaaaaaaaaaaaaaaaaaaa
```

取消时必须再次提供完全相同的 Job ID，避免粘贴或输入错误导致其他任务被取消。脚本
需要稳定结构化输出时，可以添加 `--format json`。

## 理解常见状态

- **排队 / 运行 / 验证**：任务正在等待、传输或检查结果。
- **等待认证**：通过 OpenSSH 更新或完成认证，然后恢复任务。
- **等待冲突处理**：选择覆盖、跳过或自动重命名。
- **已暂停**：持久进度会被保留；准备好后按 `U`。
- **等待重试**：可以安全重试的传输错误正在等待下一次尝试。
- **已完成但来源保留**：目标已经验证，但移动操作无法证明可以安全删除来源，因此
  有意保留。
- **失败 / 已取消**：开始新任务前，请检查 Job 并刷新两端。

传输速度使用短时间移动平均值，只在数据流动时显示。目录操作在部分阶段可能还无法
确定总字节数。

## 中断后的恢复

daemon 会在写入过程中记录安全进度。重启后，它会重新检查来源、部分目标和最终
目标，再决定是否继续。任何不确定情况都会明确暂停或失败，不会靠猜测处理。

暂停通常会保留可以继续使用的部分数据。如果删除会损失恢复证据，取消或失败也可能
保留 Job 自己的部分文件，或编辑过程的安全副本。排查 Job 时不要手动删除
`.part`、备份、SQLite 或 WAL 文件。

如果移动操作在目标提交后、来源删除前中断，来源会留在原处。检查两边后再恢复。

daemon 或持久 Job 状态不健康时，请查看[恢复指南](../help/recovery.md)。

## 远端到远端传输

两个 SSH 端点之间的复制使用两条普通 OpenSSH/SFTP 会话，并通过本地 daemon 的
受限内存中继。它不会：

- 把密钥、票据或 Agent 复制到服务器；
- 开启 Agent forwarding 或 GSS 凭据委托；
- 放宽主机密钥策略；
- 把完整传输文件保存到预览/编辑缓存。

公开构建不提供生产环境的服务器端加速或服务器间直传。标准 SFTP 始终是受支持的
通道。
