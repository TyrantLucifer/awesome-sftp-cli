# Stage 2 — Durable Transfers

- **状态**：Not Started
- **阶段类型**：持久化写操作与数据安全
- **前置条件**：[Stage 1 — Read-only Explorer](01-read-only-explorer.md) 已通过退出门禁
- **完成后进入**：[Stage 3 — Preview, Edit & Cache](03-preview-edit-cache.md)

## 1. 阶段目标

把 Vim 操作意图转化为可审查、可持久化、可暂停、可恢复的后台文件作业。覆盖本地↔本地、本地↔远端、同一远端和远端 A↔远端 B；在尚无 Helper 与直传的情况下，跨远端默认由本机守护进程以有界内存中继。

本阶段的首要原则是：任何中断都不能把不完整目标伪装成成功文件，任何跨端点移动都不能在目标验证前删除源。

## 2. 范围

### 2.1 包含

1. 内部操作剪贴板与 `y`（复制来源）、`d`（剪切来源）、`p`（在目标创建操作意图）。
2. 操作规划器、能力快照、冲突预检和高风险确认。
3. SQLite 持久化的 Job、步骤、检查点、事件和结果摘要。
4. 后台工作队列、并发限制、暂停、继续、取消、失败重试和守护进程重启恢复。
5. 本地/远端文件与目录复制、移动；远端 A↔B 的有界内存中继。
6. 临时目标、验证、原子提交和跨端点移动的延迟源删除。
7. 覆盖、跳过、重命名、应用到本作业剩余冲突等冲突决策。
8. 基础变更：建目录、重命名和通过独立 `D` 动作触发的显式删除，遵循能力与确认策略。
9. Jobs 视图所需的最小状态和进度数据；完整抽屉体验留给 Stage 3。

### 2.2 非目标

1. 不做跨主机直传；远端 A↔B 使用本地中继。
2. 不安装 Helper，不依赖远端 hash、快速 walk、trash 或 copy-data 等增强。
3. 不提供图片预览、外部编辑/打开或离线固定缓存。
4. 不承诺内容级去重或块级增量同步。
5. 不把 `d` 解释成立即删除；实际删除必须是独立、显式且高风险的动作。

## 3. 核心不变量

### 3.1 阶段依赖

- Stage 1 已稳定提供守护进程生命周期、版本化 IPC、LocalFS/SFTP Provider、认证 Broker 和双栏 Location 模型。
- 写操作必须扩展共享 Provider 契约，不能通过临时 shell 命令绕开系统 OpenSSH、结构化错误和取消语义。
- Stage 0/1 的 Fake Provider、临时 `sshd`、断线和认证夹具可直接用于写入与恢复测试。

### 3.2 数据安全不变量

以下不变量优先于性能和便利性：

1. **计划冻结**：作业排队后保存来源、目标、策略、能力快照和用户决策；运行中不得因 UI 当前目录变化而改变语义。
2. **临时写入**：文件内容先写入同目标目录的唯一 `.part-<job>` 临时名；最终名只在验证后提交。
3. **不覆盖未确认目标**：目标存在或在执行中出现时进入 `waiting_conflict`，除非冻结计划已有适用策略。
4. **移动是复制后删除**：跨文件系统或跨 Endpoint 移动先完成复制、验证和提交，再单独执行源删除。
5. **不确定时保留源**：验证强度不足、提交结果不确定或源删除失败时，作业以明确结果结束并保留源。
6. **重试只发生在幂等阶段**：对提交、重命名和删除等结果可能已生效的操作，先检查后置条件再决定下一步。
7. **取消不等于删除证据**：默认保留可安全续传的临时文件和检查点；用户可显式清理。
8. **内存有界**：传输缓冲受配置预算约束，与文件总大小无关；目录递归按流式队列推进。

## 4. 详细交付物

### S2-D01 操作意图与规划器

- `y`/`d` 保存 Location 引用、选择快照和来源类型，不预先下载内容。
- `p` 生成 Operation Intent，由守护进程解析目标、递归范围、冲突和能力。
- 规划器输出可序列化计划：路径映射、预计项、路由、验证策略、风险和需确认事项。
- 普通无冲突复制可直接入队；覆盖、不可恢复删除或影响范围不确定的操作必须展示计划并确认。
- 规划时不可穷举百万树；允许先产生“发现阶段”，但范围根、策略和安全预算必须冻结。

### S2-D02 作业数据模型与状态机

规范状态至少包括：

`draft → awaiting_confirmation → queued → running → verifying → completed`

可从适用状态进入：

- `paused`
- `waiting_auth`
- `waiting_conflict`
- `retry_wait`
- `failed`
- `canceled`
- `completed_with_source_retained`

每次转换记录原因、时间、单调事件序号和负责步骤。非法转换被拒绝并记录诊断；重启后从数据库和实际文件后置条件共同恢复，不能只信内存状态。

### S2-D03 SQLite 持久化

- 保存 Job、不可变计划、可变状态、条目/步骤、检查点、冲突决策和有限事件历史。
- 数据库只保存文件操作元数据，不保存密码、SSH 私钥、Askpass 响应或 Kerberos ticket。
- 所有状态转换与关键检查点使用事务保证一致性。
- 建立 schema 版本，为 Stage 6 迁移保留正向升级路径。
- 数据库损坏或迁移失败时停止写操作，保留原库和诊断；不得创建一个空库后假装无作业。

### S2-D04 执行器与有界中继

- Worker 数量、每作业并发和总缓冲都有上限。
- Local↔SFTP 和 SFTP A↔SFTP B 使用相同的流式复制语义。
- 远端 A↔B 不落完整文件到本地磁盘；守护进程使用固定大小缓冲在两个会话间搬运。
- 进度以已确认写入/检查点为基础，不能用仅已读取字节冒充完成。
- 连接失效时作业进入等待/重试状态，不阻塞其他 Endpoint 的无关作业。

### S2-D05 临时目标、续传与提交

- 临时文件与最终目标位于同一目标目录，以提高最终重命名原子性。
- 检查点记录可验证的偏移、目标临时文件身份和来源指纹。
- 续传前重新检查来源、目标临时文件和能力；任何不一致都不得盲目追加。
- 基线验证使用实际可用的大小、读取结果和元数据；强 hash 在 Stage 4 可增强。验证不足时按策略重新读取或保留源。
- 提交通过目标 Endpoint 支持的原子重命名完成；若服务器不保证原子性，计划和结果必须显式标示降低的保证。
- 最终目标已出现时停止提交并进入冲突处理，不直接替换。

### S2-D06 冲突处理

冲突类型至少包括：

- 目标已存在。
- 目标在规划后被创建或修改。
- 来源在规划/传输期间改变。
- 目标大小写、Unicode 规范化或文件/目录类型冲突。
- 权限、空间或父目录状态改变。

用户决策为：覆盖、跳过、自动改名、重试、取消；“应用全部”只作用于当前 Job 和匹配的冲突类别。覆盖先保留安全提交边界；如果能力不足以安全替换，必须显示风险并再次确认。

### S2-D07 移动与删除

- 同一 Endpoint 且能力明确支持时，可用原子重命名作为移动快路径。
- 其他移动一律执行复制→验证→提交→删除源。
- 源删除是独立可审计步骤；失败不会回滚已验证目标，而是返回 `completed_with_source_retained`。
- `d` 只表示剪切来源；`D` 才创建实际删除意图，二者不能因键位状态或重复命令而混淆。
- 显式删除优先使用 Endpoint 宣告的 trash 能力；纯 SFTP 基线没有可靠 trash 时，必须标为不可恢复并二次确认影响范围。
- 递归删除先生成边界明确的计划，拒绝根目录、空/歧义 Location 或超出 Endpoint 根的路径。

### S2-D08 进度、控制与最小 Jobs 视图

- TUI 能看到作业状态、来源/目标、已处理项、字节、当前条目、等待原因和最近错误。
- 暂停停止获取新工作，并在安全检查点收敛；继续从重新验证后的检查点恢复。
- 取消停止未提交工作，保留/清理临时文件按明确策略执行。
- 守护进程无 TUI 时继续执行非交互步骤；需要认证或冲突决策时进入等待。
- 日志事件可通过 Job ID、步骤 ID 和 Endpoint ID 关联。

## 5. 里程碑顺序

### M2.1 — 持久化状态机骨架

1. 建立 SQLite schema 和迁移入口。
2. 实现 Job/步骤状态机、事务转换和事件流。
3. 用 Fake Provider 覆盖重启、重复事件和非法转换。

门禁：守护进程在任意状态被终止后，重启能给出确定且安全的下一状态。

### M2.2 — 单文件复制与提交

1. Operation Intent 和计划冻结。
2. 本地↔本地、本地↔远端的临时写、验证、提交。
3. 冲突和基础暂停/取消/重试。

门禁：短写、断线和 `kill -9` 后最终目标从不呈现部分内容。

### M2.3 — 目录与双远端

1. 流式目录发现和有界任务队列。
2. 远端 A↔B 本地内存中继。
3. 每项冲突和作业内 apply-all 策略。

门禁：大树处理不全量入内存，任一 Endpoint 断开只暂停相关步骤。

### M2.4 — 移动、删除与恢复闭环

1. 同 Endpoint 重命名快路径与跨 Endpoint 移动安全序列。
2. 显式删除、能力感知 trash 和不可恢复确认。
3. 守护进程重启、升级前停机和临时文件清理策略。

门禁：所有故障注入下均满足核心不变量，且可从 Jobs 视图解释结果。

## 6. 可验证退出标准

- [ ] `y`/`d`/`p` 对单项、多选和目录产生稳定、可预览的 Operation Intent。
- [ ] 四类路径（本地↔本地、本地↔远端、同远端、远端 A↔B）复制通过。
- [ ] 远端 A↔B 中继不把完整文件落本地，峰值缓冲受配置上限约束。
- [ ] 最终目标只在验证和提交后出现；失败时临时文件不会被列为成功结果。
- [ ] 覆盖、跳过、自动改名、应用全部和执行中竞态均有确定行为。
- [ ] 暂停、继续、取消、认证等待、冲突等待和失败重试可跨守护进程重启恢复。
- [ ] 跨 Endpoint 移动在目标验证/提交前绝不删除源。
- [ ] 验证不足或源删除失败时状态为 `completed_with_source_retained`，且用户可定位两份文件。
- [ ] `kill -9`、断网、短写、权限、磁盘满和陈旧 `stat` 故障矩阵通过。
- [ ] SQLite 中无认证秘密，损坏/迁移失败会安全停止写操作。

## 8. 失败与回滚策略

- 任一写入错误先停止该条目并保存上下文；默认不删除临时文件或源文件。
- 直至 Stage 5 都不尝试未经预检的跨主机直传；双远端失败回到等待/重试，不换成不可审计 shell 命令。
- 数据库事务失败立即停止领取新步骤；保留数据库、WAL 和日志用于恢复。
- schema 迁移失败恢复原文件，守护进程只读诊断，不自动重建。
- 提交响应丢失时先检查最终名、临时名和指纹，再判定成功、重试或人工冲突。
- 删除结果不确定时按“可能已删除”处理并重新 `stat`；不得无条件重复递归删除。
- 若新版本发现安全不变量缺陷，关闭写操作能力并允许只读浏览；回退二进制前确保旧版本理解当前 schema，或使用已验证的数据库备份。
