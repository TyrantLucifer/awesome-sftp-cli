# 下一阶段路线图

本路线图描述内部预览发布后的优先方向，不是已批准的逐任务实施计划。任何新增行为都应先形成独立设计或 ADR，再进入实现。

## 1. 内部预览反馈闭环

优先处理真实用户能感知的问题：安装、首次 SSH 连接、双栏浏览、一次安全传输、远端编辑、Job 恢复、诊断与回滚。每个问题必须有可复现路径、明确失败语义和定向回归测试。

严格 `X.Y.Z` 公开预览使用 checksum 验证的一键安装脚本、Homebrew tap 与 `amsftp upgrade` 渠道感知编排降低安装/升级成本；自动升级仍需在原生 macOS/Linux 上持续演练活动 Job 拒绝、daemon 恢复、回滚与撤回。该预览不替代 Apple 签名、公证或公开 1.0 门禁。

完成标准：新用户只依赖 README 和用户文档即可完成安全首次运行；主要失败类别能通过 `doctor`、稳定错误码或脱敏支持包定位。

## 2. 兼容性与运行稳定性

持续验证受支持 macOS/Linux、OpenSSH、SFTP server、终端、文件系统和升级/回滚场景。优先修复会造成数据不一致、任务不可恢复、凭据泄露或无界资源占用的问题。

完成标准：支持声明与真实原生证据一致；兼容性退化有清晰降级原因和恢复动作。

## 3. Production Helper

只有在离线签名密钥托管、产物验签、撤销、安装同意、版本高水位、远端安装原子性和最终发行物绑定全部有真实证据后，才能讨论开放 production Helper。

完成标准：安全评审和公开发行门禁明确批准；标准 SFTP 在 Helper 缺失、损坏或不兼容时仍完整可用。

## 4. Production Level 2

Level 2 必须使用真正的进程/网络隔离数据面，不能以同进程 fixture 或控制面预检代替。任何直传方案都不得复制密钥、默认转发 Agent、默认委托 Kerberos或绕过 host-key 策略。

完成标准：隔离数据面、故障降级、提交边界、结果等价、完整性与大文件资源证据全部通过独立设计和验证。

## 5. 公开 1.0

公开发行需要最终候选在受支持平台完成原生验证、签名、公证、可复现构建、SBOM/provenance、干净安装、升级、回滚和发布撤回演练。

完成标准：[`RC-GATES.md`](../release/RC-GATES.md) 的公开发行检查项全部关闭；README、功能矩阵、兼容性和发布渠道与同一最终字节身份一致。

## 不在近期范围

- Windows 原生体验、GUI 或 FUSE 挂载。
- 自研 SSH/Kerberos 协议栈或凭据保险箱。
- 未经用户同意的远端部署、提权或常驻服务。
- 以削弱数据安全和恢复语义为代价的性能捷径。
