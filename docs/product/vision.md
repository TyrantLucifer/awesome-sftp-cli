# Vim-first 双栏 SFTP Commander：产品愿景

> - 产品名：`AMSFTP`（公开命令 `amsftp`；仓库名 `awesome-sftp-cli`）
> - 文档状态：1.0 产品愿景；内部预览已发布，公开发行尚未完成
> - 目标版本：1.0
> - 最后更新：2026-07-19

## 1. 为什么要做

开发工作已经迁移到远端 Linux 开发机，但文件浏览、预览、编辑和传输仍然割裂：终端里的 `ssh`/`sftp` 适合执行单点操作，桌面 SFTP 客户端又很难完整继承本机 OpenSSH 的 Kerberos/GSSAPI、`~/.ssh/config`、`ProxyCommand`、跳板机和 agent 环境。远端文件用本机应用打开时，还会遇到缓存、回传、并发修改和失败恢复等问题。

本产品要把这些动作收敛到一个 Vim-first、双栏、终端原生的文件指挥器中。它不替代 OpenSSH，也不试图把远端文件系统伪装成本地磁盘；它复用用户已经验证可用的 SSH 身份链路，在其上提供结构化浏览、可靠传输、快速预览和可恢复后台任务。

## 2. 一句话愿景

让 macOS 和 Linux 的终端用户像使用 Vim 与经典双栏文件管理器一样，自由操作本地及任意远端文件，同时保留 OpenSSH 的全部认证能力，并获得可恢复、可解释、不会悄悄破坏数据的传输体验。

## 3. 核心用户

首要用户是习惯 Vim、终端和 OpenSSH 的开发者，包括：

- 本机为 macOS 或 Linux，主要工作目录位于一台或多台 Linux 开发机上的用户；
- 使用 Kerberos/GSSAPI、SSH agent、密钥、密码、多因素认证、`ProxyCommand` 或 `ProxyJump` 的用户；
- 需要在本地、本地与远端、两个不同远端之间频繁浏览和搬运文件的用户；
- 需要用 Vim 快速查看或修改远端文件，也偶尔要用本机默认应用打开图片、PDF、表格等文件的用户；
- 处理数万项目录、百万文件树、数百 GB 文件，不能接受整树载入内存或前台会话断开即丢任务的用户。

产品首先服务作者本人和同类 Vim 用户：默认体验必须不依赖鼠标、不要求理解内部实现，也不能为了照顾图形界面习惯而削弱 Vim 操作的一致性。

## 4. 产品承诺

### 4.1 Vim-first，而不是只提供几组 Vim 快捷键

默认处于 Normal 模式，`h/j/k/l` 负责层级与光标移动，`Tab` 切换活动栏，`v`/`V` 建立连续选择，`Space` 增减离散标记。`y` 复制引用、`d` 剪切引用、`p` 在目标栏创建传输意图；真正删除使用大写 `D` 并明确确认。首版支持计数前缀与 `.` 重复，但不实现完整 Vim 宏和命名寄存器。

`e` 优先进入终端编辑器；编辑器解析顺序为显式配置、`$VISUAL`、`$EDITOR`、`nvim`、`vim`、`vi`。`o` 才使用本机默认应用。默认键位可配置，但出厂配置始终以 Vim 用户为第一优先级。

### 4.2 两栏都是完整端点

任一栏都可以是本地目录、远端 A 或远端 B，并可独立切换端点。产品必须平等支持 local/local、local/remote、remote/local、同一 remote 的两个路径以及 remote A/remote B，不把“左边本地、右边远端”写死。

启动方式同时覆盖直接路径、已保存工作区和交互式选择：

```text
amsftp ~/Downloads devbox:/data/output
amsftp --workspace release
amsftp
```

无参数启动时，从 `~/.ssh/config` 中提取可连接主机并模糊选择；工作区保存两个端点、路径和视图设置，但绝不保存密码、私钥或 Kerberos 票据。

### 4.3 OpenSSH 是唯一认证事实来源

远端 SFTP 会话由系统 `ssh` 启动 SFTP subsystem，产品通过其标准输入输出承载结构化 SFTP 协议。Kerberos/GSSAPI、host key、agent、密钥、密码、多因素认证、跳板与代理规则均由用户现有 OpenSSH 配置决定，产品不维护第二套 SSH 配置解析和认证实现。

交互认证通过受控的 askpass broker 转交当前附着的 TUI；没有可交互客户端时，任务进入 `waiting_auth`，而不是失败循环或把凭据写入磁盘。daemon、数据库、缓存和日志均不得持久化认证秘密或票据。

### 4.4 普通操作快，高风险操作可见

普通复制在按下 `p` 后立即排队；覆盖、递归删除、跨端点移动、直传等高风险动作必须展示可理解的执行计划或进入冲突确认。`d` 只表示“剪切待粘贴”，不会立即删除源文件。

传输写入目标端临时 `.part-<job-id>`，完成校验后再原子重命名。跨端点移动严格执行“复制、验证、删除源”；只要验证证据不足，源文件就保留，并以 `completed_with_source_retained` 明确收尾。任何情况下都不得在目标提交前删除源。

### 4.5 任务属于后台 daemon，而不是某个界面进程

一个按用户运行的本地 daemon 持有连接、工作区、任务、缓存和持久化状态。TUI 是可随时附着或退出的客户端；退出 TUI 不终止正在运行的任务。客户端与 daemon 通过权限为 `0600` 的 Unix socket 和版本化消息协议通信。

任务状态、阶段检查点和冲突决策写入 SQLite。网络中断、daemon 重启和客户端退出后，任务可以从安全边界恢复；只有幂等阶段可自动重试，提交、重命名和源删除等非幂等阶段必须先检查远端后置状态。

### 4.6 零部署基线，可选 helper 增强

只要系统有 OpenSSH 并且远端提供标准 SFTP，核心浏览、传输、恢复和有限搜索就必须可用。远端 helper 不是前置条件，只在用户明确批准后安装，用于高速遍历、内容搜索、强哈希、磁盘空间、tail/watch、同机复制和直传预检。

helper 采用按用户安装的版本化可执行文件，并服从两次确认：先按当前 policy 完成 Ed25519 验签、key/artifact 撤销、签名目标/协议/`min_client`、release floor 与兼容检查，再展示 preliminary mapping/probe consent；取消时必须产生零次 probe。获准后才做 fresh OS/架构/uid/home namespace probe、目标与安装 `(version, hash)` high-water 检查、本地产物校验并派生只读安装计划，再展示 final actual-plan consent；取消时 application-managed install tree 必须保持零 create/content。最终获准后才上传临时文件，上传前后均校验 SHA-256，设为 `0700`，最后 no-replace 原子安装并完成协议/能力握手。它不监听端口、不常驻、不提权；崩溃、缺失或版本不匹配时必须安全降级到 SFTP。

### 4.7 大规模不是事后优化

目录列表、递归遍历、搜索结果和远端中继都采用流式、分页或有界缓冲。目标工作负载为单目录数万项、整树数百万文件、单文件数百 GB；任何正常操作都不能因为规模增长而要求把完整目录、完整树或完整文件放入内存。

## 5. 核心体验

### 5.1 主界面

主界面固定为两个平等文件栏和一个可展开的底部抽屉。顶部或栏头展示端点、规范化路径、连接/能力状态；文件区支持增量加载、排序、过滤和多选；底部状态行展示当前模式、剪贴板、任务摘要和可执行提示。

底部抽屉有三个主要页签：

- **Preview**：文本、代码、JSON、元数据、图片能力检测和外部预览器结果；
- **Jobs**：队列、速度、进度、当前阶段、等待原因、重试与冲突动作；
- **Log**：连接、路由、降级、校验和错误的结构化事件。

预览默认有界读取；大文本支持头部、尾部和继续加载。图片按 Kitty、iTerm2、Sixel 等终端能力显示，能力不足时清楚降级为元数据或外部打开。

### 5.2 编辑与本机打开

远端文件编辑采用“缓存租约”流程：下载或复用内容缓存，记录打开时的远端版本，暂停 TUI 并启动编辑器，退出后判断本地内容是否变化，再重新检查远端。若两侧都变化，必须提供差异、另存为、跳过或显式覆盖选择；绝不静默覆盖。

`o` 将远端文件置于受租约保护的本地缓存后调用 macOS `open`、Linux `xdg-open` 或用户配置的 opener。外部应用退出或文件变化后的回传同样走任务、校验和冲突流程。

### 5.3 搜索与命令

- `/`：当前目录即时过滤，由本地 daemon 完成；
- `f`：递归文件名搜索，优先 helper，缺少 helper 时使用有预算的 SFTP 遍历；
- `g/`：递归内容搜索，优先 helper 的安全 `rg` 或内置扫描器，缺少 helper 时进入明确标记的受限慢速模式；
- `!`：在当前栏位置执行一次性本地或远端命令；
- `gs`：暂停 TUI，进入当前栏对应的本地或远端交互 shell，退出后恢复并刷新。

产品不嵌入终端模拟器；命令和 shell 复用用户现有终端与 OpenSSH。

## 6. 传输路由与能力阶梯

每个操作先生成不可变的执行计划，再由任务引擎运行：

1. **Level 0：标准 SFTP**。提供浏览、基础文件操作、流式复制、断点、客户端有界中继和有限递归搜索；这是永远可用的兼容基线。
2. **Level 1：可选 helper**。增加快速遍历、内容搜索、强哈希、磁盘空间、tail/watch 和同主机复制。
3. **Level 2：直传**。仅在源/目标 helper、网络可达性、目标空间、源主机已有的非交互认证和用户策略全部通过预检后启用。

同端点操作优先原子 rename、服务端 copy-data 或 helper 同机复制。remote A 到 remote B 默认通过本机 daemon 进行有界内存中继且不落盘；满足全部 Level 2 条件时可直传。产品不会为了直传自动开启 agent forwarding、Kerberos delegation、复制私钥或绕过 host key 检查。直传在目标提交前失败时可以带原因降级为本机中继；提交边界之后不盲目改路由。

## 7. 安全与一致性底线

以下约束属于产品契约，而非实现建议：

- 不保存密码、私钥、agent 内容、Kerberos 票据或交互认证答案；
- 不绕过系统 OpenSSH 的 host key 与连接策略；
- 不默认转发 agent、不默认委托 Kerberos、不复制密钥到远端；
- 不把未完成内容直接暴露为最终目标文件；
- 不在目标验证和提交完成前删除移动源；
- 不静默覆盖并发变化的远端文件；
- 不把 helper 变成远端常驻服务或开放网络监听；
- 不把完整远端文件作为跨主机中继的本地落盘前置条件；
- 不吞掉错误或只显示“失败”，每个失败都要包含端点、阶段和可行动原因；
- 日志和诊断导出必须默认脱敏。

纯 SFTP 端点若没有 trash 能力，删除必须明确标记为不可逆。取消传输默认保留可恢复的 part 与检查点，并向用户提供清理动作；磁盘空间或安全策略不允许保留时，必须在执行前说明。

## 8. 本地缓存策略

默认缓存是有配额的短期 LRU，按内容去重，并通过租约避免正在预览、编辑或外部打开的文件被清理。工作区可选择：

- **Ephemeral**：会话结束后按策略回收；
- **Pinned offline**：显式固定，可离线读取，重新连接后按版本规则校验；
- **Default LRU**：在全局与工作区配额内自动淘汰。

缓存保存的是文件内容和必要的版本元数据，不保存认证材料。daemon 异常退出后要能回收过期租约、识别孤儿临时文件并保持被固定内容完整。

## 10. 成功标准

1. 使用现有 Kerberos/GSSAPI 与复杂 `~/.ssh/config` 的用户无需把凭据搬进产品即可浏览远端。
2. 用户只靠 Vim 默认键位即可在任意两个端点间完成浏览、选择、预览、复制、移动和安全删除。
3. TUI 退出、daemon 重启、网络中断、认证过期、磁盘满和短写后，任务要么安全恢复，要么停在可解释、可操作的状态，不产生伪装成成功的损坏文件。
4. 远端编辑和本机打开不会静默覆盖他人的并发修改。
5. 缺少 helper 时核心功能仍然成立；helper 故障只损失增强能力。
6. 单目录 50,000 项、百万文件树和 100 GB 稀疏文件测试中，内存保持有界，界面与取消操作保持可响应。
7. 每项 1.0 能力都能从稳定功能 ID 追溯到阶段、验收标准、自动或人工证据以及当前状态。

## 11. 1.0 明确不做

- Windows 原生体验或以 Windows 为主的兼容承诺；
- 原生 GUI、Finder/文件管理器扩展或 FUSE/系统盘挂载；
- 云端同步服务、多用户协作平台或跨设备状态同步；
- 自研 SSH/Kerberos 协议栈或独立凭据保险箱；
- 内嵌终端模拟器；
- 完整 Vim 宏、命名寄存器和插件生态；
- 未经用户批准的远端软件安装、提权或常驻服务；
- 以默认 agent forwarding、Kerberos delegation 或复制私钥换取直传便利。

这些边界不否定未来扩展，但任何扩展都不能破坏 OpenSSH 单一认证来源、双栏端点平等、任务持久化和数据安全四项核心承诺。
